# Yaksok — CONVENTIONS (문서 시스템 규칙)

> 이 repo의 md 문서들이 "작업할 때마다 알아서 최신 상태로 유지"되도록 하는 규칙.
> 각 문서 헤더의 `(CONVENTIONS §N)`은 이 파일의 절 번호를 가리킨다.
> 세션 시작 시 `CLAUDE.md`(→ `@AGENTS.md`)가 이 규칙을 참조하도록 연결돼 있다.

## §1 문서 지도 (무엇이 어디에)

| 문서 | 역할 | 언제 읽나 |
|---|---|---|
| `AGENTS.md` (= `CLAUDE.md`) | 스택·협업·검증 + **이 문서 시스템의 진입점** | 세션 시작 시 항상 |
| `HANDOFF.md` | **현재 구현 상태 + 다음 할 일** (상태의 단일 원본) | 작업 이어받을 때 가장 먼저 |
| `LOG.md` | 날짜별 굵직한 변경 한 줄 | 무엇이 언제 바뀌었나 |
| `PLAN.md` | 목표·범위·기술 결정·마일스톤 | "무엇을 / 왜" 볼 때 |
| `README.md` | 소개·실행법·환경변수 (외부/신규 관점) | 프로젝트를 처음 받았을 때 |
| `CONTRIBUTING.md` | Git 초보 팀원용 협업 루틴 | Git 작업하기 전 |

## §2 진실의 원천 + 갱신 대원칙

- 같은 사실이 여러 곳에 있으면 **우선순위**: 구현 상태 = `HANDOFF` · 목표/결정 = `PLAN` · 실행법 = `README`.
- **코드와 문서가 어긋나면 코드가 정답** — 발견 즉시 해당 문서를 고친다.
- 갱신은 *현재 상태가 정확하도록 덮어쓴다*. 변경 이력을 길게 쌓지 않는다 (그건 git log · `LOG` 몫).

## §3 LOG.md

- 양식: 날짜 헤더(`## YYYY-MM-DD`) 아래 **굵직한 변경 한 줄**.
- 기록 대상: 기능·범위·스택·배포처럼 "남이 알아야 할" 변경. 자잘한 fix·refactor·docs는 git 커밋 메시지로 충분.
- 언제: 그런 변경을 머지/완료한 세션을 끝내기 전.

## §4 HANDOFF.md — frontmatter = 상태의 단일 원본

- 상단 frontmatter(`status` / `updated` / `summary` / `repo`)가 **프로젝트 현재 상태의 단일 출처**.
- `updated`는 갱신한 날짜(`YYYY-MM-DD`), `summary`는 한 줄 현황.
- "지금 상태"를 묻는 모든 곳은 이 frontmatter를 기준으로 한다.

## §5 HANDOFF.md — 본문은 매 세션 갱신

- **작업 세션을 끝낼 때마다** 본문 3블록을 현재에 맞게 갱신한다:
  - **마지막 작업** — 이번 세션이 한 일 (요약).
  - **다음 할 일** — 바로 착수 가능한 항목. 사람이 직접 해야 하는 건 `(사람)`으로 표시.
  - **막힌 것** — 블로커. 없으면 "해당 없음".
- 테스트: 다른 PC/다른 사람이 `HANDOFF`만 읽고 상태와 다음 할 일을 파악할 수 있는가?

## §6 PLAN.md vs README.md (역할 분리)

- **PLAN** = 목표 · 범위(포함/제외) · 기술 결정 · 마일스톤. "무엇을, 왜."
- **README** = 소개 · 실행법 · 환경변수. "어떻게 돌리나." (외부/신규 관점)
- 결정이 바뀌면 `PLAN`을, 실행 절차가 바뀌면 `README`를 고친다. 둘을 섞지 않는다.
